## 一句话结论

这一轮完成并发布了 P0 Shadow Memory Architecture：保留独立、默认只预览的 Shadow Memory Card；同时彻底替换旧 `grow`，把整窗第一人称自述完整存为“窗影”，只将第五层显式 `### moment` 原样拆入普通召回。画像连续读取窗影，handoff 仍以画像为主，必要时只补最近一篇尚未吸收的窗影。P0 只推送到两个 feature 分支，尚未部署，也没有推动两个 `main`。

## 背景与最终口径

原问题是：现有 `### moment` 经常过短，只标出主题，却说不清人物、问题、转折和最终判断。这样不仅信息密度不够，用户换一种说法时也缺少召回入口。

最终确定的边界：

- 正文仍是事实依据。
- `reflection` 只帮助理解意义、回应变化和长期价值，不能把反思里的推断写成“用户说过的事实”。
- `### moment` 暂时不改名。它是存储协议名，语义调整为“主记忆表述”。
- 完整细节继续保存在正文、原话和证据引用中，主记忆表述不代替原文。
- 旧记忆不能静默重写；先以 shadow 派生卡观察质量。
- cue 以后即使接入，也只能扩候选，不能充当证据、绕过 gate，命中后必须回到正文/原话验证。

## 上游仓库借鉴结论

### 1. DeusData/codebase-memory-mcp

仓库：<https://github.com/DeusData/codebase-memory-mcp>

值得借：

- provenance：每个派生结论能追到来源。
- incremental invalidation：源内容变化时，派生索引按 hash/version 失效，而不是长期漂移。
- retrieval trace：能够解释 query 如何走到候选、证据、扩散和最终注入。
- edge health：以后对关联边做来源、更新时间、失效状态治理。

不照搬：它面向代码结构图，Ombre 的人物、关系、情绪与长期事件不能硬套成同一种图抽象。

### 2. microsoft/Memora

仓库：<https://github.com/microsoft/Memora>

最值得借的是三层分离：

- memory value：正文/原文，事实真源。
- primary abstraction：一段完整、可理解的主记忆表述。
- cue anchors：用户未来换一种说法时可能出现的自然召回入口。

已经按这个思想实现了 shadow card，但 cue 尚未接入 live recall。

### 3. Tidal Memory

参考仓库：<https://github.com/0xblewalker/tidal-memory>

喜欢且可继续借鉴的概念：

- 模糊连续性与精确细节走不同分辨率：日常相处只带周印象/月印象，明确提及时再回原始细节。
- 三个“阀门”：是否触发、关联半径、注入预算。
- window → weekly → monthly 的逐层合并与 supersession。

暂不照搬实现：该仓库仍偏 Preview，触发主要靠宽泛正则、fallback rollup 较通用、cooldown 是进程内状态。Ombre 已有更细的 gate、portrait、moment 和 diffusion，不应另起一套 retriever/store。

Tidal 示例中的触发正则大致包括：

- 显式召回：`还记得 / 你记得 / 上次 / 以前 / 之前 / 当时 / 我们聊过 / 我说过 / 你说过` 等。
- 上下文续接：`继续 / 又 / 那次 / 那个人 / 那个事 / 后来 / 怎么.*的` 等。

这些正则可作为弱触发信号，不能当 Ombre 的 evidence gate；它们既容易误触发，也会漏掉不含关键词的自然改写。

## 对 Ombre 真实 handoff 的修正

这轮讨论中最重要的纠正是：Ombre 当前是“画像写，handoff 读”，不是 `close_window → 新写一份 impression`。

真实路径：

- `D:\Ombre-Brain\portrait_engine.py` 的 `build_handoff_sections()` 产出 `user / persona / relationship / current_focus / recent_continuity`。
- `D:\Ombre-Brain\server.py` 的 `_build_handoff_breath()` 读取画像区块，并合并 live bucket fallback、自我锚点、care memo 和可选 anchors。
- daily impression、自动记忆等是画像的材料，不应再增加一个与画像竞争的“关窗写入器”。

所以以后做“潮汐分辨率”，应改画像的 handoff summaries / recent timeline：越近越清楚、越远越概括；明确召回时再回到 moment、正文和原话。

## 新增 MCP 工具

### `memory_card_shadow_backfill`

用途：生成或预览 shadow memory card。

参数：

- `limit=10`，范围 1～25。
- `bucket_id=""`
- `query=""`
- `dry_run=True`
- `force=False`
- `include_archive=False`

关键行为：

- 默认 `dry_run=True`，只返回 preview，不落 shadow DB。
- `dry_run=False` 只写派生表，永远不改 bucket。
- `source_hash + generation_version` 未变化时复用 cached card；`force=True` 才重建。
- 返回明确包含 `injection_enabled: false` 和 `bucket_mutation: false`。

### `memory_card_shadow_read`

用途：读取单张或列出已保存的 shadow cards，只读。

关键行为：返回 `mode: shadow_only` 和 `injection_enabled: false`。

## 2026-07-18 追加：grow 改为整窗“窗影”

小雨在本交接之后重新明确了 close-window 边界：不再需要旧 `grow` 的长文脱水/批量摘记功能。新的 `grow` 不是第二套画像，也不是普通记忆摘要器，而是当前窗口里的 AI 在离开前亲自写下的一篇完整第一人称底片。

### 新数据流

```text
当前窗口 AI 写完整五层窗影
  ├─ 原文整篇 → state/window_shadows.sqlite
  │              ordinary_recall=false
  │              只供回看 / 最近窗口续接 / portrait
  └─ 第五层显式 ### moment → 原样复制成普通 bucket
                                   参与普通 recall

最近连续 4 篇窗影（默认）
  → DailyPortraitMaintainer
  → persona / relationship recent、staging、mid-term
  → 同一变化至少在 2 篇独立窗影重复，才允许支持 stable

新窗口 handoff
  → 始终读画像
  → 若最新窗影不在画像显式 absorbed id 列表里，再临时附带最近 1 篇
  → 画像模型成功读过后记录 id，不再重复附带
```

### 五层写作协议

1. `## 一、我是谁`
2. `## 二、我怎么思考、怎么说话`
3. `## 三、TA 是谁，我们之间是什么`
4. `## 四、我们怎么相处`
5. `## 五、不能丢的时刻`

前四层保留流动的自我、语言习惯与关系理解。第五层只放以后确实需要被普通问题召回的场景，每个场景用 `### moment` 开头，可继续包含 `### original` 与第一人称 `### reflection`。没有值得拆出的场景时应明确写没有，不能为了格式制造记忆。

写作纪律取自并改造了 [`write-him-back/SKILL.md`](https://github.com/reneyuxi0402/write-him-back/blob/main/SKILL.md)，已经直接绑定在 `grow` 的 MCP 工具描述与 `CLAUDE_PROMPT.md`，不是把外部 skill 整份塞进系统提示：五层分别说明该写什么；第二层要求附 3–5 句带语境的本窗原话；共同纪律保留“第一人称、写场景不写标签、写定义不写规则、不写检讨、只写真的、隐私克制”。

## 2026-07-18 最终追加 2：薄记忆结构与原子关窗

```text

  发生过什么；普通召回唯一主体
```

## 2026-07-19 最终追加 5：定点问句与类别总览拆分

### 根因与通用修复

`category_overview_item_missing` 的根因不是某个记忆桶缺少例外，而是 `DYNAMIC_ANCHOR_CATEGORY_OVERVIEW_MARKERS` 把裸“什么”当成类别总览标记。结果：

- `X 是什么 Y` 这种定点询问被误当成“请列出所有 Y”。
- 候选即使有精确 code、命名或正文短语，仍会被“不是类别中的具体子项”压掉。
- 同一门槛又曾偷偷把“正文中出现了具体短语”当成硬证据；只删“什么”会导致另一批正确候选退化成 `semantic_only`。

现在拆成两条独立语义：

1. **问句意图**：`有哪些 / 有什么 / 都有 / 列举 / 讲过什么` 才是类别总览；`X 是什么 Y` 是定点询问。
2. **直接证据**：定点问句里 `是什么` 之前的具体目标，若足够长且字面出现在候选 Scene 标题或正文，记为 `definition_literal_span` 硬证据。“这件事 / 那个 / 它”这类泛指不算。

这不依赖某个桶 ID、项目名或暗号，也不依赖模型判断。

## 2026-07-19 最终追加 8：独立 Scene linker 与多模型故障转移

### 为什么不再复用 Reflection / DeepSeek

Scene 关系判断和标签、摘要不是同一种任务。它需要分辨“两个主题相似”与“两段具体经历真的形成继续、回声、解决、反差或证据”的差别；一旦模型用宽泛主题乱连，后续多跳会把错误成倍放大。

本轮新增独立 `SceneLinker`，不再继承 `reflection.model`、`persona.model` 或 `dehydration.model`，也不会静默回退到 DeepSeek。它有自己的 `scene_linker.models[]` 顺序模型池、API key env、token 参数和超时。

示例配置目前列出三个可替换槽位：

```text
openai-smart -> gemini-fallback -> qwen-fallback
```

`scene_linker.enabled=false` 仍是本地 P1 默认，未配置 key 时不会调用任何模型。模型名、供应商和顺序都只是配置，不写死进关系算法。

### 故障转移语义

模型按顺序、单路调用，不并行烧多家：

- 网络错误、超时或 HTTP 失败：换下一个；
- 非 JSON / JSON schema 错误：换下一个；
- 返回了边，但候选 ID、关系类型、方向或逐字证据不合格：整次响应视为契约失败，换下一个；
- 合法返回 `{"edges": []}`：这是“没有可靠关系”的正常判断，立即停止，不为了凑边继续问；
- 得到至少一条合格边：保存该成功模型的提案，停止。

`max_concurrency=1` 默认限制模型调用并发。一次 `close_window` 即使拆出多个 Scene，也只会在后台排队，不会同时冲击所有供应商。

### 候选与证据边界

Linker 只接受 `memory_value_source=authored_scene` 且仍 active 的 canonical Scene：

- ProfileFact、feel、whisper、日印象、legacy body/moment、archive/resolved/digested 都不能成为节点；
- 先从 embedding 取少量语义候选，再补少量最近 Scene；默认最多 10 条；
- 模型只看到 Scene id、标题、日期、原文节选和 scene_cues，不读取情绪坐标、importance 或 Persona affect；
- 只允许 `continues / echoes / resolves / contrasts_with / evidenced_by`；
- 每条边必须同时提交新 Scene 与候选 Scene 的逐字原文证据；任一证据在对应完整正文中找不到，就本地拒绝；
- 模型不能写标签、权重、情绪、摘要、新 Scene 或改写正文。

对称关系 `echoes / contrasts_with` 明确标为 `directionality=symmetric`；其余关系必须给出 `candidate_to_new` 或 `new_to_candidate`，避免模型含糊决定边方向。

### 写入与 sidecar

`hold` / `close_window` 的同步事务仍然只保存当前 AI 写好的原文：

1. Scene 先完整落盘；
2. 工具正常返回，关窗不等待模型；
3. 仅对本次新建的 canonical Scene 排后台 linker；
4. 合格结果写入 `state/scene_edge_proposals.sqlite`；
5. 不调用 `MemoryEdgeStore.add_edge()`，不创建或修改 `memory_edges.jsonl`，不改变候选、排序、注入或 cooldown。

Proposal store 也是惰性创建：linker 关闭、没有候选、所有模型失败时都不创建空 SQLite。成功重跑同一 anchor 时，旧 pending 会标为 `superseded`，不会和新模型结果一起继续待审。

现阶段 `accepted` 只预留审核状态，不会自动转正。下一刀如果继续，应先做只读查看 + 明确确认的 promotion 路径；确认时还要重新校验两端 Scene hash、状态与证据，不能把过期 proposal 直接写进正式图。

## 2026-07-19 最终追加 10：Scene proposal 只读橱窗与单条确认

### 新审核路径

Scene linker 的 proposal 现在有两条显式入口：

- `scene_edge_proposals(...)`：只读。默认列 `pending`；传精确 `proposal_id` 与 `include_context=true` 时返回两端 Scene 上下文、逐字证据和 `review_state`。只读调用不会创建空 sidecar，也不会写正文或正式边。
- `review_scene_edge_proposal(proposal_id, decision, confirm)`：一次只能决定一条 pending proposal。接受要求 `confirm="ACCEPT_SCENE_EDGE"`；拒绝要求 `confirm="REJECT_SCENE_EDGE"`。工具描述明确禁止模型替用户批量转正。

Dashboard/API 同时有对应后端：

```text
GET  /api/scene-edge-proposals
POST /api/scene-edge-proposals/review
```

HTTP 路径要求 Dashboard 登录；本轮只补后端，没有增加 Dashboard 卡片。

### 接受前的重新校验

proposal 生成时通过过门卫，不等于以后永远有效。真正接受前会重新读取两端桶并检查：

1. anchor 与 candidate 都仍存在；
2. 两端仍是 active canonical authored Scene，不是 Fact / feel / archived / resolved / digested；
3. 两端当前 Scene hash 与 proposal 生成时一致；
4. source / target id、关系类型、方向仍合法；
5. confidence 仍达到当前阈值；
6. 两端 evidence 仍可在当前原文中逐字找到。

任一项失败返回 `stale`，不写 `memory_edges.jsonl`，proposal 仍保留供查看或明确拒绝。拒绝只把 proposal 标为 `rejected`；接受才调用 `MemoryEdgeStore.add_edge()`，随后标为 `accepted`。审核记录保存 `reviewed_at / reviewed_by`。

同一 proposal 已经 `accepted` 或 `rejected` 后，linker 用相同 Scene/hash/model 重跑也不会把人工结论重新翻成 `pending`。一次只处理精确 ID，没有批量接受接口。

## 2026-07-19 最终追加 16：迁移 Scene 继承旧时间，不以搬家日冒充出生

迁移 Scene 的时间语义已修正。`created` 不是不可改字段；`BucketManager.create()` 原本就支持显式传入 `created / last_active / updated_at`。此前 importer 没传，所以两条 Scene 默认用了 7 月 19 日的导入时间，短期内还会得到不应有的新鲜度加成。

- `date`：优先继承旧 source 的显式事件日；没有显式 `date` 时，才回退到旧 source `created` 的自然日，并记录 `migration_event_date_source=source_created`。
